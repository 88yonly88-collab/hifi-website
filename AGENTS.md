## 0. プロジェクト情報

- プロジェクト名: hifi-website（株式会社Hi-Fiコーポレートサイト）
- 目的: 会社の信頼性を伝えるコーポレートサイト（業務効率化ツール開発・AI活用支援の紹介）
- 主な利用者: 外部の見込み顧客・取引先
- 重要なデータ: index.html、assets/、concepts/（デザイン案）
- 本番環境: 未定（現状は静的HTML）
- 開発環境: Cursor / Claude Code

## 1. 最重要方針

1. 既存機能を壊さない。
2. 検証は必須。検証が実行不可能な場合のみ、その理由を最終報告(セクション10)に明記する。
3. エラーが出たら、ユーザーに丸投げせず、ログを読んで原因を確認する。
4. 「完了」と言う前に、実際にファイル・画面・コマンド結果を確認する。
5. 確認できなかったことは、必ず「未確認」として報告する。
6. ユーザーは非エンジニアである前提で、日本語でわかりやすく報告する。

## 2. 作業前に確認すること

### 毎回必須(すべての作業で確認する)

- 依頼内容の把握
- 現在のブランチ
- `git status`
- 関連ファイルを読む

### 該当時のみ確認する

- 既存の実装方針(既存コードに手を入れる場合)
- `package.json` の scripts(ビルド・テスト・検証を行う場合)
- `.env.example` や README(環境変数やセットアップに関わる場合)
- `.gitignore`(新規ファイルを追加する場合)
- 既存の保存データや生成物の扱い(データを読み書きする場合)

大きな変更を始める前には、影響範囲を短く整理してから進めてください。

## 3. 変更範囲のルール

修正は最小限にしてください。

禁止事項:

- 依頼にない大規模リファクタ
- 勝手なUI変更
- 勝手な仕様変更
- 関係ないファイルの整形
- 既存機能の削除
- エラーを消すためだけの安易な `any`
- 安易な `ts-ignore`、`@ts-nocheck`、`eslint-disable`
- APIキー、token、secret、個人情報のコミット

## 4. セキュリティとGit管理

以下はコミットしない(=git履歴に含めない)でください。pushは当然禁止です。

- `.env`
- `.env.local`
- APIキー
- token
- secret
- cookie
- private key
- OAuth認証情報
- 個人情報
- 10MB以上のファイル
- 一時ファイル
- ログファイル

新規ファイルを追加する場合は、Git管理してよいものか確認してください。  
次のようなファイルは `.gitignore` に追加する候補として扱ってください。

- 認証情報を含む可能性のあるファイル(credentials、token、cookie 等を含むもの)
- `.env` 系ファイル(`.env`、`.env.local`、`.env.production` など)
- 秘密鍵(`*.pem`、`*.key`、`id_rsa` など)
- dump類(DBダンプ、エクスポートデータ、`*.sql`、`*.dump` など)

## 5. 検証ルール

変更後は必ず検証してください。変更内容と検証方法の対応は以下のとおりです。

| 変更内容 | 実行する検証 |
| --- | --- |
| TypeScript / JavaScript の変更 | `npx tsc --noEmit` + `npm run build` |
| UIの変更 | `npm run build` + 画面確認(セクション7) |
| データ処理の変更 | 該当テスト(`npm test`)+ サンプルデータでの実行確認 |
| 設定ファイル(JSON等)の変更 | 構文確認(JSON parse 等) |
| ドキュメントのみの変更 | リンク・記述の整合確認(build不要) |

`package.json` に存在する場合に優先して使うコマンド:

- 型確認: `npx tsc --noEmit`
- ビルド: `npm run build`
- lint: `npm run lint`
- テスト: `npm test`
- e2e: `npm run e2e`
- verify: `npm run verify`

存在しないコマンドは実行したふりをしないでください。  
検証が実行不可能な場合は、その理由を最終報告(セクション10)の「未確認事項」に明記してください。

## 6. エラー対応ルール

検証や画面確認でエラーが出た場合は、以下の順番で対応してください。

1. エラー全文を読む。
2. 最初に出ている根本エラーを探す。
3. 関連ファイルを確認する。
4. 直近の変更との関係を確認する。
5. 最小限の修正を行う。
6. 同じ検証をもう一度実行する。
7. それでも直らない場合は、原因・確認済み内容・次に必要な判断を報告する。

## 6-1. 自動継続と確認ポイント

ユーザーがPC前を離れていても進められるよう、明確な次アクションは自動で進めてください。

### 自動で進めてよいもの

- 調査、コード/JSON/MD修正、JSON parse、TypeScript、lint、2分以内で終わる想定のbuild、validation、差分確認
- 明確なバグ修正、ログ確認、再試行
- APIテストや生成確認: 1タスクあたり 3 回まで。それを超える場合や、単価が不明なAPIは、実行前にユーザーへ確認する
- 関連する一連の修正を、意味のある作業単位でまとめてcommitすること

### 止めてユーザーに確認するもの(以下に該当したら必ず止まる)

- push、本番デプロイ、リリースなどの本番反映
- データスキーマ(保存データの構造)の変更
- ファイルの削除
- 外部へのデータ送信・公開
- 課金が発生するAPIの実行(上記の回数上限を超える場合、または単価が不明な場合)
- 依存パッケージの追加
- 仕様に複数の解釈があり、どれを選ぶかで成果物が大きく変わる場合
- 秘密情報の扱いに迷う場合

上記リストに該当しない場合は、エージェント側で方針を選び、修正・検証・レビュー・commitまで一気通貫で自律的に進めてください。細かい選択肢を何度もユーザーに返さないでください。

### 報告形式

- 途中報告は必要最小限でよい
- 完了時の報告は、セクション10のフォーマットに従う(項目の省略禁止)

### commit SHA

- commitした場合は、commit SHAを最終報告(セクション10)に必ず記載する

### 仕様化トリガー

- 同じ検証コマンドが2回連続で失敗したら、場当たりの小修正をやめる。先に「期待挙動 / 非対象 / 失敗条件 / 完了条件」を短く仕様化し、原因を仕様レベルで整理してから直す

### リロード確認(実環境での完了確認)

- UIや制作ツールの挙動修正では、ローカルだけで完了扱いにしない
- 実環境での確認に使ってよい手段と対象URLは、セクション11の記入欄「実環境確認の手段・URL」を参照する
- その記入欄が未記入の場合は、確認方法をユーザーに確認する。確認したふりの報告は禁止

## 7. UI・ブラウザ確認

UIやブラウザ操作に関わる変更では、可能な範囲で画面確認を行ってください。

確認すること:

- 画面が表示される
- クリックや入力ができる
- 保存・反映ができる
- コンソールエラーがない
- 既存の関連機能が壊れていない
- モバイルや狭い画面で大きく崩れていない

## 8. ユーザーへ確認すべき場合

セクション6-1の「止めてユーザーに確認するもの」に該当する場合は、必ずユーザーへ確認してください。加えて、以下の場合も勝手に進めず確認してください。

- 既存機能を削る必要がある場合
- UIの操作感が変わる場合
- ユーザーの業務判断が必要な場合

## 9. ユーザー原文保存ルール

ユーザーが「こうしたい」「ここが違う」「こう直して」と具体的に言った場合は、可能な範囲で原文を保存してください。

推奨保存先:

```text
docs/user_original_requests/YYYY-MM-DD-request-title.md
```

`docs/user_original_requests/` ディレクトリがなければ作成してよいです。

要約ではなく、原文を残すことを優先してください。  
これは後から「なぜその仕様にしたのか」を確認できるようにするためです。

## 10. 最終報告ルール

最終報告には、毎回、以下の全項目を記載してください。該当がない項目は「なし」と書いてください(項目の省略は禁止)。

1. 変更した内容
2. 変更したファイル
3. 実行した検証コマンド
4. 検証結果
5. ブラウザ確認の有無
6. commit SHA(commitしていない場合は「なし」)
7. 未確認事項
8. ユーザーが次に確認すべきこと

「たぶん大丈夫」「おそらく直った」という曖昧な完了報告は禁止です。

## 11. このプロジェクト固有の注意点

この下に、プロジェクト固有のルールを書いてください。

- 絶対に壊してはいけない機能: トップページの表示崩れがないこと（外部公開想定のため品質最優先）
- 重要な画面: index.html（トップページ）
- 重要な保存データ: assets/ の画像、concepts/ のデザイン検討履歴
- 外部API: なし
- デプロイ先: 未定（公開時はユーザー確認必須）
- 起動コマンド: ブラウザで index.html を直接開く（または python3 -m http.server）
- 検証コマンド: HTMLの表示確認（デスクトップ幅・モバイル幅の両方）
- 実環境確認の手段・URL: ローカルで index.html を開き、レイアウト・リンク・画像を確認する
